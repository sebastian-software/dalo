# v0.4.0 Release Notes

Date: 2026-07-05
Changelog: [`0.4.0`](../../CHANGELOG.md)

`v0.4.0` hardened Dalo's sync, adoption, catalog, and instruction-pack paths. The release keeps the V1/V1.1 surface intact, but it changes a few upgrade behaviors that matter for existing `0.3.x` stores.

## Upgrading from 0.3

Run `dalo status` or `dalo doctor` before the first `dalo sync` after upgrading. Fix any reported conflicts first, then run `dalo sync --dry-run` to inspect planned target changes before applying them.

### Adoption replacement is explicit

`dalo adopt --yes <skill>` no longer replaces the original target entry. Use:

```sh
dalo adopt <skill>
dalo adopt --replace <skill>
```

or the resolve helper:

```sh
dalo resolve adopt --replace <id>
```

The global `--yes` flag is reserved for future prompts and is currently a no-op. It never implies `--replace`.

### Team sources may be pinned

`0.3.x` team sources did not always record an update policy. In `0.4.0`, sources only update during `sync` when they are explicitly configured to track. If a team source should continue following its branch, edit the source entry in `config.toml`:

```toml
update_policy = "track"
```

Then run:

```sh
dalo sync --dry-run
```

### Slot names are stricter

Skill slot names now reject uppercase names, leading or trailing dots, path-like names, and Windows device basenames such as `con` or `nul`. Rename the skill directory upstream or in the local source, then approve the renamed skill if needed.

### Approvals are source-qualified

Bare approvals such as `review` no longer approve every skill with that name. Re-approve with the source-qualified stable ID shown by `dalo status`, for example:

```toml
[[approvals]]
scope = "skill"
value = "company:review"
```

### Config schemas reject unknown fields

Dalo now rejects unknown fields in persisted config, lock, source-lock, state, and approval files. This catches typos and unsafe downgrade paths earlier. If a future Dalo version writes a new field, older `0.4.x` binaries may refuse to read it rather than guessing.

## Notes for Contributors

Future breaking changes should use Conventional Commit breaking syntax, for example `feat!:` plus a `BREAKING CHANGE:` footer, so Release Please can surface the upgrade note automatically.
