# Release audit — 2026-07-04 (v0.3.0 → v0.4 plan)

Full audit of the codebase, test suite, CI, documentation, and website at
HEAD `55b2313` (v0.3.0). Every finding below is tracked as a GitHub issue;
the release plan is the epic
[#85](https://github.com/sebastian-software/dalo/issues/85).

## Scope and method

- All 19 `src/` modules read in full, with six review lenses: core
  correctness (resolver/store/materialize/lockfile/inventory/source), CLI and
  operational modules, performance, reliability under failure, test
  coverage, and documentation/website accuracy.
- Build verification on this checkout: `cargo build`, `cargo clippy
  --all-targets --all-features -- -D warnings`, and `cargo fmt --check` are
  all clean. `cargo test`: 113/114 unit tests pass — the one failure is
  environmental (permission-based failure injection does not work as root,
  [#70](https://github.com/sebastian-software/dalo/issues/70)).
- Distribution checks: `dalo` is not published on crates.io (the name is
  free), releases carry no binary artifacts, and no `LICENSE` file exists
  despite `license = "MIT"` in `Cargo.toml`.

## Overall assessment

The codebase is careful in many places: atomic TOML writes, TOCTOU
re-checks at apply time, conservative conflict handling, id validation
against path traversal, deterministic resolution (including a permutation
test), and a store lock with dead-owner reclaim. The design docs are
unusually thorough for a project at this stage.

The gaps cluster where failure meets the safety guarantees. The core
promises — never delete unmanaged files, never rewrite unmarked
instruction content, `--dry-run` never mutates, approval as the trust gate
— all hold on the happy path and all have at least one reachable violation
under errors, races, or crashes. That is the theme of the v0.4 plan: make
the guarantees hold under failure, then make the project installable and
presentable.

Performance is healthy at target scale (~10 sources / ~200 skills). The
pure-Rust work is milliseconds; user-visible cost is git subprocesses.
Only two hot-path items made the plan (duplicate sync pipeline, duplicate
catalog scan+hash).

## Findings by area

### Safety-critical correctness (Phase 1)

| Issue | Finding |
| --- | --- |
| [#44](https://github.com/sebastian-software/dalo/issues/44) | A failed or degraded source scan makes `sync` silently remove every symlink of that source |
| [#45](https://github.com/sebastian-software/dalo/issues/45) | `instructions enable/disable` treats any read error as an empty file and overwrites user content; writes are non-atomic |
| [#46](https://github.com/sebastian-software/dalo/issues/46) | `--dry-run` is ignored by `instructions enable/disable`, `source select`, `resolve keep` |
| [#47](https://github.com/sebastian-software/dalo/issues/47) | The materializer removes foreign (user-created) symlinks at recorded link paths |
| [#48](https://github.com/sebastian-software/dalo/issues/48) | Skill-scope approvals match bare slot names, widening trust across sources |
| [#49](https://github.com/sebastian-software/dalo/issues/49) | Crash mid-sync/adopt leaves store-pointing symlinks permanently blocked as "foreign" |
| [#50](https://github.com/sebastian-software/dalo/issues/50) | Git subprocesses have no timeout and can prompt for credentials while holding the store lock |

### Reliability hardening (Phase 2)

Marker injection bricking managed blocks
([#54](https://github.com/sebastian-software/dalo/issues/54)), missing
instruction-block drift detection
([#55](https://github.com/sebastian-software/dalo/issues/55)), the adopt
replace delete-before-link window
([#52](https://github.com/sebastian-software/dalo/issues/52)), the doctor
advice loop for real-entry slots
([#53](https://github.com/sebastian-software/dalo/issues/53)), the store
lock stale-reclaim race
([#51](https://github.com/sebastian-software/dalo/issues/51)), plus eight
further medium findings
([#56](https://github.com/sebastian-software/dalo/issues/56)–[#63](https://github.com/sebastian-software/dalo/issues/63)).

### Tests and CI (Phase 3)

Coverage of the happy paths and safety guarantees is good (~170 tests, all
isolated via tempdirs). Untested: corrupt/truncated TOML, kill mid-sync,
concurrent processes, missing git binary, the `--` URL injection guard,
user edits inside managed blocks, unicode/case-collision skill names
([#70](https://github.com/sebastian-software/dalo/issues/70)–[#74](https://github.com/sebastian-software/dalo/issues/74)).
CI lacks dependency audit (`serde_yaml` is deprecated/unmaintained),
coverage measurement, and an MSRV contract
([#73](https://github.com/sebastian-software/dalo/issues/73)).

### Performance (Phase 4)

`sync` builds the status/lock pipeline twice, doubling git spawns
([#68](https://github.com/sebastian-software/dalo/issues/68));
`source select` walks and re-hashes the whole catalog twice
([#69](https://github.com/sebastian-software/dalo/issues/69)).

### Polish batches (Phase 5)

Low-severity verified fixes grouped for one PR each:
core ([#64](https://github.com/sebastian-software/dalo/issues/64)),
instructions ([#65](https://github.com/sebastian-software/dalo/issues/65)),
catalog ([#66](https://github.com/sebastian-software/dalo/issues/66)),
CLI/doctor ([#67](https://github.com/sebastian-software/dalo/issues/67)).

### Distribution and documentation (Phase 6)

Missing `LICENSE`
([#75](https://github.com/sebastian-software/dalo/issues/75)), no
crates.io publish and no release binaries
([#76](https://github.com/sebastian-software/dalo/issues/76)), stale
status docs describing the pre-0.2 product
([#77](https://github.com/sebastian-software/dalo/issues/77)), missing
user reference / CONTRIBUTING / SECURITY / troubleshooting / man pages
([#78](https://github.com/sebastian-software/dalo/issues/78)–[#81](https://github.com/sebastian-software/dalo/issues/81)).

### Website (Phase 7)

Factual drift against the shipped product
([#82](https://github.com/sebastian-software/dalo/issues/82)), messaging
that undersells the differentiator
([#83](https://github.com/sebastian-software/dalo/issues/83)), and UX
upgrades ([#84](https://github.com/sebastian-software/dalo/issues/84)).

## Suggested v0.4.0 cut

Phases 1–3 plus #75/#76/#77/#82: "the release where the guarantees hold
under failure, and you can `cargo install dalo`." Everything else can
trail into v0.4.x without blocking.
